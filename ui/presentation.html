Context

The problem for AI + store teams

Before

  • Agents invent raw /wp-json/wc/v3/… paths
  • OAuth 1.0a signing is easy to get wrong
  • No shared schema language between models
  • One-off scripts per automation
  • Secrets leaked into prompts & notebooks

After (woo-mcp-server)

  • Purpose-built tools: woo_products_list, …
  • Library owns OAuth / Basic Auth / retries
  • Zod schemas = machine-readable contracts
  • One STDIO binary for any MCP client
  • Env-only credentials, fail-fast at startup

The MCP server is a thin orchestration layer. It does not rewrite the HTTP stack — it calls woocommerce-rest-ts-api typed methods as-is.

Step 1 of 5 · Setup

Install & build the monorepo

# clone & enter
git clone https://github.com/Yuri-Lima/woocommerce-rest-api-ts-lib.git
cd woocommerce-rest-api-ts-lib

# workspace install (root library + packages/mcp-server)
pnpm install

# build library (CJS+ESM) and MCP server
pnpm run build

# unit + integration tests (hermetic nock; live suite skipped by default)
pnpm --filter woo-mcp-server test
Workspace dependency: MCP package uses "woocommerce-rest-ts-api": "workspace:*" so it always links the local library — never a stale published copy during development.

Step 2 of 5 · Setup

Configure credentials (fail-fast)

If any required var is missing, the server exits immediately with a copy-pasteable help message. No silent fallbacks.

VariableRequiredDefaultPurpose
WC_URLyes—Store base URL
WC_KEYyes—Consumer key ck_…
WC_SECRETyes—Consumer secret cs_…
WC_VERSIONnowc/v3API version prefix
WC_RATE_LIMIT_PER_SECONDno5Outbound throttle
WC_QUERY_STRING_AUTHnofalsetrue on some HTTPS hosts
 
export WC_URL=https://mystore.com
export WC_KEY=ck_xxxxxxxx
export WC_SECRET=cs_xxxxxxxx
# HTTP local Docker stores: leave query-string auth OFF (use OAuth)
export WC_QUERY_STRING_AUTH=false

Functionality tour · 5

Quality guarantees for every tool

  • AI-friendly descriptions — what + when/why (≥ 2 sentences)
  • Zod input — per-field descriptions & examples
  • Zod output — response validated before return
  • Pagination — page/per_page + total/totalPages/currentPage
  • Structured errors — MCP tool errors, not HTTP dumps
  • Nullish param stripping — WC 10.x rejects empty enums (fixed live)

Use case 01 · Customer support

Support copilot in Claude Desktop

Persona: Support lead · Goal: answer “Where is my order?” without logging into wp-admin

Conversation pattern

  1. Read woo://store/info for currency/context
  2. woo_customers_search with email fragment
  3. woo_orders_list filtered by customer
  4. woo_orders_get for line items & status
  5. woo_orders_notes_create to log the interaction
User: "Order for jane@acme.com stuck?"

Agent plan:
  woo_customers_search { query: "jane@acme.com" }
  woo_orders_list { customer: 12, status: "processing" }
  woo_orders_get { id: 101 }
  → "Order #101 is processing, 2 items,
     total $49.98, placed 2024-06-01."
  woo_orders_notes_create {
    order_id: 101,
    note: "Customer contacted via chat"
  }

Use case 02 · Merchandising / ops

Low-stock inventory check

Persona: Ops engineer · Goal: weekly restock list grouped by category

Best path

Invoke the built-in prompt template:

prompts/get
  name: inventory-check
  arguments:
    threshold: "5"
    include_outofstock: "true"

The prompt instructs the model to page products, list variations when needed, group by category, and suggest reorder qty.

Tools under the hood

  • woo_products_list + stock_status
  • woo_variations_list for variable parents
  • woo_categories_list for labels
  • Optional: woo_reports_top_sellers to refine qty
Output: markdown table SKU · Name · Category · Stock · Suggested reorder · Product ID

SEO & product quality audit

Use case 05 · Data engineering

Bulk catalog import / price sync

Persona: Integration developer · Goal: sync 200 SKUs from ERP without 200 round-trips 
woo_products_batch {
  "create": [
    { "name": "SKU-A", "regular_price": "10.00" },
    { "name": "SKU-B", "regular_price": "12.00" }
  ],
  "update": [
    { "id": 10, "regular_price": "31.00" }
  ],
  "delete": [ 999 ]
}

Why batch matters

  • WC REST supports batch endpoints natively
  • Respects WC_RATE_LIMIT_PER_SECOND better than N creates
  • Same pattern: woo_orders_batch for mass status updates
Always dry-run on a staging/TasteWP/Docker store first — batch deletes are permanent when force is true.

Use case 06 · Platform engineering

Embed MCP in your own agent runtime

You are not limited to Claude Desktop. Any MCP client can connect over STDIO.

import { Client } from "@modelcontextprotocol/sdk/client/index.js";
import { StdioClientTransport } from "@modelcontextprotocol/sdk/client/stdio.js";

const transport = new StdioClientTransport({
  command: "node",
  args: ["./packages/mcp-server/dist/cli.js"],
  env: {
    ...process.env,
    WC_URL: process.env.WC_URL!,
    WC_KEY: process.env.WC_KEY!,
    WC_SECRET: process.env.WC_SECRET!,
  },
});

const client = new Client({ name: "my-agent", version: "1.0.0" });
await client.connect(transport);

const tools = await client.listTools();
const result = await client.callTool({
  name: "woo_products_list",
  arguments: { per_page: 5 },
});

In-memory transport is used in our Jest suite for the same protocol surface without spawning a process.

Decision guide

When to use the library vs the MCP server

NeedUse library directlyUse woo-mcp-server
Backend service / worker ✓ typed TS imports optional
AI agent / Claude / GPT tools — ✓ designed for this
Fine-grained control of every request ✓ via purpose-built tools
Prompt playbooks & resources — ✓
OAuth signing only ✓ core value inherits automatically

Both live in the same monorepo. MCP never forks the HTTP layer.
